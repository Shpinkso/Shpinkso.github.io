I\u2019ve never been completely sold on Cyclomatic Complexity as a metric, it maps to linearly independent paths through code which may be useful to get an indication on the level of testing an application needs, assuming code coverage is not available (another metric that needs to be treated with caution), but how much value is that? Most applications are multi threaded and do not run in a linear fashion, I\u2019d argue that there\u2019s considerably more value in measuring the readability of code in order to allow people to understand what it does and how to change it. I\u2019ve seen easy-to-read and hard-to-read methods that evaluate to the same value on the Cyclomatic Complexity scale. Perhaps that\u2019s ok and we care about logical complexity, but my (weakly held) strong opinion is that the real value is in ease of human understanding when it comes to code.<\/span><\/p>

Perhaps Cyclomatic Complexity has a place as an indicator of worsening or improving code complexity, but I\u2019m happy to say that I have found something that comes a lot closer to evaluating \u201ccan I work out what it\u2019s doing?\u201d type complexity, in my mind at least.<\/span><\/p>

As part of some work that we\u2019re doing on metrics, I had a look into alternative code complexity measure and stumbled upon Cognitive Complexity – a metric that SonarQube, amongst others presumably, can report.<\/span><\/p>

Now before moving on, I\u2019m assuming that there\u2019s little disagreement that \u201cinternal code quality\u201d (of which readability is an aspect) is a valuable thing. If not, stop reading and go here: <\/span>https:\/\/martinfowler.com\/articles\/is-quality-worth-cost.html<\/span><\/a><\/span>\u00a0<\/span><\/p>

So what is Cognitive Complexity? Other than being obviously how hard something is to get your head around. Well there\u2019s a great whitepaper <\/span>here<\/span><\/a><\/span>, but I\u2019ll try to summarise the content below.<\/span><\/p>

How is this done? There are three basic rules<\/span><\/p>
Ignore Structures that allow multiple statements to be readably shorthanded into one.<\/span><\/b><\/h4>
A good example of this is the breaking up of code into methods to increase readability. As a result, Cognitive Complexity does not increment for methods as Cyclomatic Complexity does.<\/span><\/p>
The aim here is to not penalise good programming practice like an increased number of smaller methods, because it is likely to reduce complexity for a reader.<\/span><\/p>
Increment when flow-breaking structures are nested<\/span><\/b><\/h4>
Nested breaks in flow increase count as a proportion of how deeply they are nested. So a new \u201cif\u201d that is nested 4 deep adds 5 to the complexity score. 1 for the \u201cif\u201d and 4 for the nest level.<\/span><\/p>
This is because it\u2019s intuitively easier to follow a linear list of code breaks than an equally, Cyclomatic Complexity-wise, complicated nested set of code breaks.<\/span><\/p>
Increment for each break in the linear flow of the code<\/span><\/b><\/h4>
Anything that breaks the top to bottom, left to right flow of the code increases Cognitive Complexity, similarly to Cyclic Complexity, but there are some interesting differences.<\/span><\/p>
1.\u00a0\u00a0\u00a0\u00a0 <\/span><\/span>\u201cCatch\u201d is added as a break in flow<\/span><\/p>
2.\u00a0\u00a0\u00a0\u00a0 <\/span><\/span>\u201cElse\u201d, \u201cElif\u201d and other alternatives to the initial \u201cif\u201d break in flow do increment but not with the same nesting penalty as above, as the nesting complexity is considered paid already at the initial \u201cif\u201d.<\/span><\/p>
3.\u00a0\u00a0\u00a0\u00a0 <\/span><\/span>\u201cSwitch\u201d statements only incur a single increment, regardless of cases. Switch statements are considered more understandable then multiple \u201cif\u201d\/\u201delse if\u201d blocks.<\/span><\/p>
4.\u00a0\u00a0\u00a0\u00a0 <\/span><\/span><\/span>Repeating logical comparisons with the same logic operator only increment complexity once. This is because it\u2019s fairly similar effort to understand both of the following lines<\/span><\/p>
a\n&& b<\/span><\/code><\/p>
a\n&& b && c && d<\/span><\/code><\/p>
but much harder to understand a statement with mixed logical operators<\/span><\/p>
a || b\n&& c || d<\/span><\/code><\/p>
5.\u00a0\u00a0\u00a0\u00a0 <\/span><\/span>Recursion is the one exception to the \u201cdon\u2019t increment for methods\u201d rule<\/span><\/p>
6.\u00a0\u00a0\u00a0\u00a0 <\/span><\/span>\u201cgoto\u201d, \u201cbreak\u201d and \u201ccontinue\u201d increase complexity, but no other jumps are. e.g. an early \u201creturn\u201d is considered to make code clearer so is ignored.<\/span><\/p>
The white paper referenced above is not long and I highly recommend reading it, but I hope that this is a useful summary (if nothing else, writing this caught a couple of mis-conceptions I\u2019d made from just reading the paper).<\/span><\/p>
